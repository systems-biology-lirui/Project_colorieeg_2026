从刺激图片种类到结果目录

这一页把项目里最核心的问题串起来：刺激是如何定义条件的，条件如何被转成 task 级 epoch，epoch 如何进一步变成 ROI 特征，最后又是如何进入主线 decoding、paired 分析和 HTML 报告的。

任务入口

先把刺激与任务结构想清楚

当前扩展分析几乎都以 Task1 为主，因为 Task1 同时具备四类图片和 color-gray 对照，最适合做类别 decoding 与配对比较。

Task1 的核心条件结构

默认条件顺序与 paired 分析约定一致：

0/1: face color / gray 2/3: body color / gray 4/5: object color / gray 6/7: scene color / gray

因此它既能做四分类，也能做每个类别内部的 color-vs-gray 二分类，还能做 image-paired 分析。

Task2 / Task3 的角色

它们主要参与共同通道筛选、跨任务 ROI 特征生成以及 cross-decoding。仓库里真正的配对 color-gray 方案目前集中在 Task1。

结果目录

看结果时先认目录，不要先认脚本名

processed_data

任务级 epoch 数据与元数据，是所有 ROI 特征的上游来源。与坏段剔除、repeat 重排最相关。

feature

每个 ROI 一个 `.mat` 文件。这里决定 downstream 解码用的到底是 ERP、gamma 还是 gamma multiband。

result/decoding 与 cross_decoding

主线 time-resolved decoding 结果目录。路径规则由 `newanalyse_paths.py` 统一定义。

result/reports

更偏汇总和解释性输出，包括 paired decoding 报告、metric 对照图、perm regime 对照图等。

理解这个项目最有效的方式不是“逐个文件读”，而是先分清哪些文件在制造数据对象，哪些文件在消费这些数据对象。制造端是 `Sec1` 和 `Sec2`，消费端是 `Sec3` 与 `testcode`。
