02 / Preprocess & Features

真正决定数据质量的是 Sec1 和 Sec2

如果说 decoding 和 PCA 负责“读出信息”，那么 `Sec1_preanalyse.m` 与 `Sec2_*` 负责“定义什么是可以被读出的信号对象”。这一步决定了 trial 是否保留、时间窗是否对齐、ROI 是否重叠，以及不同模态到底代表什么。

Sec1 核心逻辑

预处理不是单纯切 epoch，而是在形成可复用的 trial 轴

当前主线以 Sec1_preanalyse.m 为准，旧的 preanalyse.m 更像历史版本，不应该再作为主解释依据。

`Sec1_preanalyse.m` 的关键价值不是单纯生成两个 `.mat` 文件，而是把“哪些 trial 被保留、保留下来的 trial 在原始重复序列里对应哪里”这件事也一起保存了下来。

当前 paired decoding 会读取 `bad_epoch_indices`、`condition_original_repeat_index`、`condition_global_epoch_index` 等字段，再把 `groupedData` 对齐到实际保存下来的 repeat 轴。这是现在 color-gray 一对一比较能成立的前提。

两个预处理版本

为什么旧 preanalyse 与新 Sec1_preanalyse 结果会不一样

旧版 `preanalyse.m`

更接近早期流程，主要做 task 切分和最小 trial 数对齐。它不包含现在这一版那么完整的共享坏段剔除和 repeat 元数据追踪，因此同一 task 最终保留下来的 trial 数可能更高，但可追踪性更弱。

新版 `Sec1_preanalyse.m`

先做 robust SEEG bad epoch 判断，再对所有条件截成同一个 `min_trials`，并把剔除索引和原始 repeat 信息写入文件。结果更严格，但更适合作为 paired 或 permutation 分析的可靠上游。

ROI 映射

ROI 不是互斥箱子，而是允许重叠

映射入口

所有 `Sec2_*` 基本都依赖 get_roi_map.m，再基于 `ieegloc.xlsx` 与公共通道列表把 channel 分配给 ROI。

many-to-many 现象

同一个 channel 可能同时被放进功能 ROI 与解剖 ROI。因此两个 ROI 都显著，不代表它们来自完全独立的电极集合。

解释上的含义

跨 ROI 做比较时，需要先判断自己是否接受这种重叠。如果要互斥 ROI，就必须重写映射规则，而不是继续沿用当前输出。

Sec2 模态地图

六类 ROI 特征分别来自哪里

ERP 与 TFA 宽频分支是底层源头，gamma 家族都是在 TFA 宽频分支上继续做包络或多频带加工。

模态 代表脚本 上游来源 主要含义
ERP Sec2_1_preprocess_erp.m task*_ERP_epoched.mat 低频时域电位，主要是直接裁 ROI 和时间窗
highgamma Sec2_2_preprocess_highgamma.m task*_TFA_epoched.mat 70-150 Hz Hilbert 包络和功率
lowgamma Sec2_3_preprocess_lowgamma.m task*_TFA_epoched.mat 30-70 Hz 包络和功率
tfa Sec2_4_preprocess_tfa.m task*_TFA_epoched.mat 1-150 Hz 宽频时域 ROI 输入，不是频率 x 时间张量
gamma Sec2_5_preprocess_gamma.m task*_TFA_epoched.mat 30-100 Hz 宽频 gamma 包络，当前还未完全接入统一批处理
gamma_multiband Sec2_6_preprocess_gamma_multiband.m task*_TFA_epoched.mat 多 band、多 channel 特征，元数据最完整
预处理和特征提取的最大作用是把“原始记录”转换成“可比较对象”。一旦 trial 轴和 ROI 轴定义清楚，后面的 decoding、PCA、paired 对齐才会真正可靠。
