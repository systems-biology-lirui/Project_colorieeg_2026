05 / Code Map

要动代码时，先找到正确入口

这页不是按文件名罗列，而是按“你想做什么”来给出入口。这样后续无论你是加一个新模态、改一条批处理命令，还是补一类 paired 图，都可以先定位到该看的文件，不必在整个仓库里来回搜索。

目录职责

仓库里的主要区域各自负责什么

目录 职责 建议先看哪些文件 典型输出
newanalyse 正式主线，含预处理、ROI 特征、主 decoding、批处理和汇总 readme.md、Sec1_preanalyse.m、newanalyse_paths.py processed_data、feature、result 下的正式结果
testcode 实验性分析、方法学对照、paired color-gray 分析与审计 compare_task1_color_gray_fourclass.py、task1_paired_decoding/README.md result/reports 下的解释性图和对照输出
processed_data 被试级 task epoch 数据与 trial 元数据 关注每个 subject 下的 `task*_ERP_epoched.mat` 与 `task*_TFA_epoched.mat` 后续所有特征与 paired 重排的上游
feature ROI 特征文件仓库 按 modality/subject/roi `.mat` 查找 主线 decoding 与实验脚本的直接输入
result 正式结果、批处理输出与汇总报告 关注 `decoding`、`cross_decoding`、`reports` 子树 图、CSV、NPZ、HTML 报告

最常用入口

如果你要做 X，先打开 Y

重新生成 task 级 epoch

先看 Sec1_preanalyse.m。如果你的问题和 trial 丢失、坏段剔除、repeat 索引有关，也要先看它。

增加或检查某个模态的 ROI 特征

先看对应的 `Sec2_*`，再看 get_roi_map.m。如果涉及输出路径，再补看 newanalyse_paths.py。

修改主线 decoding 任务定义

先看 `Sec3_*` 中对应模态的主脚本，确认 `TASKS` 或任务配置里真正启用的是哪一项，而不是只看文件名。

批量跑多个被试或模态

先看 Sec4_2_batch_run_modalities.py 和 sec4_batch_config_example.json。

输出或重组 HTML 报告

先看 Sec5_2_merge_new_md_to_html.py，它负责把 Markdown 汇总成可浏览页面。

研究 color-gray 配对问题

直接从 task1_paired_decoding/README.md 进入，再按脚本决定是做 decoding、PCA trajectory 还是 cosine。

运行顺序

一个较稳妥的阅读与执行顺序

推荐执行顺序
  1. 确认 `processed_data/{subject}` 是否已有最新的 epoch 与元数据。
  2. 运行需要的 `Sec2_*` 生成对应 ROI 特征。
  3. 运行对应 `Sec3_*` 主 decoding 或 `Sec3_s*` 统计脚本。
  4. 需要批处理时改 `sec4_batch_config_example.json`，再用 `Sec4_2_batch_run_modalities.py`。
  5. 针对特殊问题，再用 `testcode` 做扩展解释或对照。
推荐阅读顺序
  1. 先理解 `newanalyse/readme.md` 对数据契约的说明。
  2. 再读 `Sec1_preanalyse.m`，确认 trial 到底如何保留。
  3. 然后读一个 `Sec2_*` 和一个 `Sec3_*`，看数据对象如何被消费。
  4. 最后再读 `testcode`，把它当作研究问题的附加镜头，而不是主流水线本体。

辅助文件

这些文件不显眼，但在维护时非常关键

newanalyse_paths.py

统一处理 feature 目录、decoding 目录、cross-decoding 目录和 reports 目录。想搞清路径时先看这里。

runtime_config.py

与批处理配置、运行时参数注入有关。配合 `Sec4_2_batch_run_modalities.py` 使用更容易理解自动跑批行为。

batch_runner_utils.py

负责批处理时的模块加载、override 应用和 runtime config 写入，适合排查批处理到底替换了哪些参数。

真正高效的维护方式不是记住所有脚本，而是记住“同一类问题总是落在哪个层级”。trial 问题回到 `Sec1`，模态问题回到 `Sec2`，主结果问题回到 `Sec3`，批量运行问题回到 `Sec4`，研究假设问题回到 `testcode`。
